📖 왜 이렇게 만들었는가? / Design Philosophy

이 앱의 본질

이 앱은 OTF Security Lab 심사용 (#21441) 데모입니다. 즉시 현장 배포용이 아니라, "왜 이런 설계가 인도주의 환경에서 필요한가"를 검증하기 위한 참고 구현체입니다.

1. 왜 단일 HTML 파일인가?

VitalGuard의 핵심 철학은 "감사 가능한 단일 파일 = 공급망 공격 면역"입니다. 외부 라이브러리·CDN·API가 0개이므로, 파일 1개의 SHA-256 해시만 검증하면 됩니다. 표준 라이브러리 의존 코드 대비 100~250배 경량입니다.

Service Worker와 Manifest는 모두 data: URL로 인라인 처리하여 단일 파일 무결성을 보장합니다.

2. 왜 계산기 위장(Plausible Deniability)인가?

난민 캠프·분쟁지역·권위주의 국가에서 활동가·기자가 디바이스를 압수당했을 때, "녹음 앱"이라는 이름만으로도 즉각적 위험에 노출됩니다. 계산기로 위장하면 검사관이 눌러봐도 평범한 계산기로 작동합니다.

비밀 트리거: 계산기 화면에서 1234 × 5678 = 입력 시 녹음 모드로 진입합니다.

3. 왜 즉시 녹음 모드가 필요한가?

현장에서 갑작스러운 폭력 사건·증거 확보 상황은 1~2초의 지연도 치명적입니다. 즉시 녹음 모드를 켜두면 PWA 클릭과 동시에 백그라운드 녹음이 시작되며, 화면에는 평범한 계산기가 표시됩니다.

4. 왜 음향 이벤트 자동 녹음인가?

총성·폭발음·비명 같은 굉음은 100~120dB의 명확한 진폭 스파이크를 보입니다. Web Audio API의 AnalyserNode로 RMS 에너지·Zero-Crossing Rate·스펙트럼 평탄도(Spectral Flatness)를 분석하여 임계값 초과 시 자동 녹음합니다. 외부 ML 라이브러리 없이 순수 DSP로 구현되었습니다.

이는 사용자가 의식을 잃거나 행동할 수 없는 위급 상황에서도 증거가 자동 보존됨을 의미합니다.

5. 왜 AES-256-GCM 암호화인가?

WebCrypto API의 PBKDF2(SHA-256, 100,000 iterations)로 비밀번호에서 키를 파생하고, AES-256-GCM(인증 암호화)로 녹음을 암호화합니다. GCM은 무결성도 함께 검증하므로 변조 시 복호화가 실패합니다.

비밀번호 없이는 NSA급 자원으로도 사실상 복구 불가능합니다. 단, 비밀번호를 잊으면 본인도 복구 불가입니다(이는 의도된 설계).

6. 왜 스테가노 파일명인가?

meeting_1735892341.webm 같은 이름은 갤러리 검색 즉시 노출됩니다. 대신 IMG_20260105_184523.jpg 형식으로 저장하면 갤러리에서 "깨진 사진"으로 보여 검사관이 무시할 가능성이 높습니다.

7. 왜 자동 청크 분할인가?

30분짜리 단일 파일은 압수 시 모든 대화가 한 번에 노출됩니다. 30초마다 분할하면 한 파일이 발견되어도 30초만 노출됩니다. 청크는 무작위 시간 간격으로 저장되어 시간 순서 추론도 어렵게 합니다.

8. 왜 패닉 와이프(3-탭 삭제)인가?

"즉시 삭제" 버튼은 검사관에게 "여기 뭔가 있다"는 단서를 줍니다. 화면 좌상단 보이지 않는 영역을 1초 이내 3번 탭하면 모든 녹음·키·캐시가 즉시 영구 삭제됩니다. 외관상 변화 없이 완료됩니다.

이는 Morgan J. 윤리 원칙 #10 — "데이터 삭제의 자유"의 직접 구현입니다.

9. 왜 메타데이터 제거인가?

WebM 컨테이너는 인코더 식별자(예: "Chrome 120 on Android"), 정확한 타임스탬프, 코덱 핑거프린트를 포함합니다. 이는 디바이스 추적·시간대 추정에 사용될 수 있습니다. 저장 직전 무작위화하여 활동가 보호를 강화합니다.

10. 왜 GDPR · UNCRC · CRPD 정렬인가?

이 앱은 설계 단계부터 다음을 보장합니다:

  • GDPR: 서버 0개 → 데이터 이전 없음 → 처리자 책임 0
  • UNCRC: 아동 보호 시 데이터 수집 없이 위험 알림만
  • CRPD: 단순 UI로 비전문가도 사용 가능

"규정 준수를 나중에 추가"가 아니라 "처음부터 위반이 불가능한 구조"입니다.

심사위원께 드리는 노트

이 데모는 현장 배포 전 단계입니다. 실제 배포 시에는 다음이 추가됩니다:

  • 독립 보안 감사 (ROS — Radically Open Security)
  • 다국어 UI (현재 한국어/영어 → 아랍어·로힝야어 등 확장)
  • 현장 파일럿 (UCL GDI Hub · IDS Sussex 협력)
  • Hippocratic License 3.0 위반 시 법적 면책 조항

모든 코드는 단일 HTML로 감사 가능하며, 외부 의존성이 0개임을 직접 검증하실 수 있습니다.

Purpose

This app is a review-stage demonstrator for OTF Security Lab (#21441). It is not for immediate field deployment, but a reference implementation to validate why such a design is needed in humanitarian contexts.

1. Why a single HTML file?

VitalGuard's core philosophy: "auditable single file = supply chain attack immunity". Zero external libraries, CDNs, or API calls — only one SHA-256 hash needs verification. 100–250× lighter than equivalent library-dependent code.

Service Worker and Manifest are inlined via data: URLs to preserve single-file integrity.

2. Why calculator camouflage (plausible deniability)?

In refugee camps, conflict zones, and authoritarian regimes, an app named "recorder" creates immediate risk upon device seizure. With calculator camouflage, an inspector tapping the icon sees a working calculator.

Secret trigger: entering 1234 × 5678 = on the calculator enters recording mode.

3. Why an instant-recording mode?

Sudden violence or evidence-capture moments tolerate no 1–2 second delay. With instant mode enabled, recording starts in the background the moment the PWA launches, while the screen displays a normal calculator.

4. Why automatic acoustic event recording?

Gunshots, explosions, and screams produce clear amplitude spikes (100–120 dB). The Web Audio API's AnalyserNode is used to compute RMS energy, Zero-Crossing Rate, and Spectral Flatness; recording auto-starts when thresholds are exceeded. Implemented in pure DSP without external ML libraries.

This means evidence is preserved automatically even when the user is incapacitated.

5. Why AES-256-GCM encryption?

WebCrypto's PBKDF2 (SHA-256, 100,000 iterations) derives a key from the password; AES-256-GCM (authenticated encryption) protects both confidentiality and integrity. Tampering causes decryption failure.

Without the password, recovery is computationally infeasible — even at NSA scale. (Forgetting it means the user cannot recover either; this is by design.)

6. Why steganographic filenames?

Names like meeting_1735892341.webm reveal intent on a quick gallery scan. Saved as IMG_20260105_184523.jpg instead, the file appears as a "broken photo" that inspectors typically ignore.

7. Why automatic chunk splitting?

A single 30-minute file exposes everything if seized. Splitting every 30 seconds limits exposure per file. Chunks are saved at randomized intervals to obscure temporal ordering.

8. Why panic wipe (3-tap)?

A visible "delete" button signals "something is here." Three rapid taps within 1 second on an invisible top-left zone permanently destroys all recordings, keys, and cache — with no visible UI change.

This directly implements Morgan J. Ethical Principle #10 — "Freedom of data deletion."

9. Why metadata stripping?

WebM containers embed encoder identifiers ("Chrome 120 on Android"), exact timestamps, and codec fingerprints — usable for device tracking and time-zone inference. These are randomized before storage to harden activist protection.

10. Why GDPR · UNCRC · CRPD alignment by design?

This app guarantees, from the design stage:

  • GDPR: zero servers → zero data transfer → zero processor liability
  • UNCRC: child-protection alerts without any data collection
  • CRPD: simple UI usable by non-specialists

Compliance is not "added later" — it is structurally impossible to violate.

Note to reviewers

This is a pre-deployment demonstrator. Production deployment will add:

  • Independent security audit (ROS — Radically Open Security)
  • Multilingual UI (currently KR/EN → Arabic, Rohingya, etc.)
  • Field pilots (with UCL GDI Hub and IDS Sussex)
  • Hippocratic License 3.0 liability clauses

All code is auditable as a single HTML file; you can directly verify zero external dependencies.
